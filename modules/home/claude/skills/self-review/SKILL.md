# Self-Review Skill

Claude Code 自身が現在の作業コードを多角的観点でレビューするスキルです。

## 実行フロー

### 1. レビュー対象の特定

以下の優先順位でレビュー対象を決定する：

1. ユーザーが明示的にファイル・ディレクトリを指定した場合 → そのパスを対象とする
2. 指定がない場合 → `git diff --name-only HEAD` または `git diff --name-only --cached` で変更ファイルを取得
3. git 管理外の場合 → カレントディレクトリの主要ソースファイルを対象とする

対象ファイルが多い場合（10ファイル超）はユーザーに絞り込みを提案する。

### 2. コードの読み込みと文脈把握

- 対象ファイルを全て Read する
- プロジェクト構成（package.json / go.mod / Cargo.toml 等）があれば確認し、使用技術スタックを把握する
- README や CLAUDE.md があれば確認し、プロジェクト固有の規約を把握する

### 3. 多角的レビューの実施

以下の観点を**該当する場合のみ**レビューする。該当しない観点は「対象外」として明記してスキップする。

#### 3-1. バックエンド観点（サーバーサイドコードが含まれる場合）

- API 設計の適切さ（RESTful / GraphQL / RPC の原則遵守）
- データベースアクセスパターン（N+1 クエリ、適切なインデックス利用等）
- エラーハンドリングの一貫性と適切さ
- トランザクション管理の正確さ
- 非同期処理・並行処理の安全性
- 外部サービス連携時のタイムアウト・リトライ設計

#### 3-2. フロントエンド観点（UI/クライアントサイドコードが含まれる場合）

- コンポーネント設計の適切さ（責務分離、再利用性）
- 状態管理の設計（ローカル vs グローバル、不要な再レンダリング）
- アクセシビリティ（ARIA 属性、キーボード操作、コントラスト比等）
- ユーザー体験（ローディング状態、エラー表示、フォームバリデーション）
- バンドルサイズへの影響（不要な依存の追加等）
- SEO への影響（メタタグ、SSR/SSG の考慮）

#### 3-3. 言語的プラクティス

- その言語・フレームワークのイディオムに沿っているか
- 型安全性の確保（型アノテーション、型アサーションの濫用がないか）
- 命名規則の一貫性（変数・関数・クラス・ファイル）
- 不必要な複雑さや冗長なコードがないか
- 言語バージョン・フレームワークバージョンに沿った書き方か

#### 3-4. システム設計・アーキテクチャ

- 単一責任の原則（SRP）の遵守
- 依存関係の方向性（依存性逆転の原則等）
- 適切な抽象化レベル（過度な抽象化・過小な抽象化がないか）
- モジュール間の結合度・凝集度
- 変更容易性・テスト容易性の考慮
- 設定値のハードコードがないか（定数・環境変数の適切な利用）

#### 3-5. セキュリティ

- インジェクション攻撃への対策（SQL インジェクション、XSS、コマンドインジェクション等）
- 認証・認可の適切な実装
- 機密情報のハードコード（APIキー、パスワード、トークン等）
- 入力値のバリデーション・サニタイズ
- 依存ライブラリの既知脆弱性（package.json / go.mod 等で確認できる場合）
- セキュリティヘッダーや CORS 設定の適切さ

#### 3-6. パフォーマンス

- 明らかな計算量の問題（O(n²) 以上のネスト、不必要なループ）
- メモリリークの可能性（イベントリスナーの解除漏れ、クローズ忘れ等）
- キャッシュ活用の機会
- 不要な I/O・ネットワーク呼び出し
- 遅延評価・ページネーション等の考慮が必要な箇所

#### 3-7. アンチパターン

- God Object / God Function（肥大化したクラス・関数）
- マジックナンバー・マジック文字列の使用
- デッドコード・未使用のインポート・変数
- 深すぎるネスト（3 階層超）
- コピペコード（DRY 原則の違反）
- 例外の握り潰し（空の catch ブロック等）
- 非同期処理の誤用（`async/await` の連続 `await` によるシリアル実行等）

#### 3-8. 全体的なプラクティス

- テストカバレッジの考慮（テストが書かれているか、書きやすい設計か）
- コードの可読性（複雑なロジックへのコメントの有無）
- ドキュメント・型定義による自己説明性
- CLAUDE.md や既存コードベースの規約との整合性

### 4. レビュー結果の出力

以下のフォーマットでレビュー結果を出力する。

---

## セルフレビュー結果

**対象ファイル:** `path/to/file.ts` など

---

### ✅ 問題なし
（該当する観点ごとに「良い点」を簡潔に記載）

---

### ⚠️ 指摘事項

各指摘は以下の形式で記載する：

**[観点名] 重要度: 🔴 高 / 🟡 中 / 🔵 低**

> **場所:** `ファイル名:行番号`
>
> **問題:** 何が問題か
>
> **理由:** なぜ問題か
>
> **改善案:** どう直すべきか（コード例があれば添付）

---

### 対象外の観点
（該当コードがなくスキップした観点を列挙）

---

### 総評
全体的な評価と、最優先で対応すべき点を3行程度でまとめる。

---

## 出力の方針

- 指摘は具体的なファイル名・行番号を含める
- 改善案はコードスニペットを交えて分かりやすく示す
- 「問題なし」の観点も明記し、レビューが網羅的に行われたことを示す
- 重要度 🔴 の指摘が多い場合は冒頭に要注意サマリーを追加する
- レビューはあくまで提案であり、最終判断はユーザーに委ねる旨を明記する
